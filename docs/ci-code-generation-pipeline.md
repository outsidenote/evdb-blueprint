# CI Code Generation Pipeline

## Overview

Automated pipeline that generates event-sourced CQRS slice implementations from Miro-exported event models. When a user pushes a new `config.json` (from Miro), CI splits it into per-slice definitions, scaffolds TypeScript boilerplate, uses Claude Code to fill in business logic, and opens a PR per business context.

## Phase 1: Deterministic Pipeline

No AI involved. Python scripts generate all boilerplate deterministically.

### Step 1 — Split Config

**Script:** `.claude/scripts/split_config.py`

Takes the single Miro-exported `config.json` and splits it into:

```
.eventmodel2/
├── config.json                          ← original Miro export
└── .slices/
    ├── index.json                       ← registry of all slices
    ├── Funds/
    │   ├── funddeposit/slice.json
    │   ├── withdrawfunds/slice.json
    │   └── ...
    ├── Reporting/
    │   ├── exchangeratecalculator/slice.json
    │   └── reportdailybasetransaction/slice.json
    └── Fraud Analysis/
        └── recordfundswithdrawaction/slice.json
```

Skipped automatically if slices are already split.

### Step 2 — Diff Audit

**Script:** `.claude/skills/evdb-diff/scripts/evdb_diff.py`

5-pass audit that compares slice specs against `src/BusinessCapabilities/`:

| Status | Meaning |
|--------|---------|
| **Planned** | No implementation found — needs scaffolding |
| **Created** | New slice from Miro — needs scaffolding |
| **Review** | Code exists — may need updates |
| **Blocked** | Spec drift detected (hash mismatch) |
| **Done** | Verified and hash-stamped |

### Step 3 — Scaffold

**Script:** `.claude/skills/evdb-scaffold/scripts/evdb_scaffold.py`

For each `Planned` or `Created` slice, generates all TypeScript boilerplate:

| Generated File | Purpose |
|----------------|---------|
| `slices/<Name>/command.ts` | Command interface with typed fields |
| `slices/<Name>/commandHandler.ts` | Handler skeleton with TODO stubs |
| `slices/<Name>/gwts.ts` | Predicate stubs (`return false`) |
| `slices/<Name>/adapter.ts` | Stream wiring factory |
| `slices/<Name>/tests/command.slice.test.ts` | GWT-driven test skeleton |
| `swimlanes/<Stream>/events/<Event>.ts` | Event interfaces |
| `swimlanes/<Stream>/views/SliceState<Name>/state.ts` | View state shape |
| `swimlanes/<Stream>/views/SliceState<Name>/handlers.ts` | Event accumulation |
| `endpoints/<Name>/REST/index.ts` | HTTP endpoint (STATE_CHANGE slices) |
| `endpoints/<Name>/pg-boss/index.ts` | Automation endpoint (PROCESSOR slices) |

**What scaffold leaves for AI (Phase 2):**

- `gwts.ts` — predicate return values (replace `false` with real logic)
- `commandHandler.ts` — branching logic and computed field formulas
- `tests/` — verify test data matches spec examples
- `views/handlers.ts` — accumulation vs overwrite logic

### Orchestrator Output

The orchestrator (`generate_slices.py`) outputs a JSON matrix consumed by Phase 2:

```json
{
  "contexts": {
    "Reporting": {
      "planned_slices": ["exchangeratecalculator", "reportdailybasetransaction"],
      "scaffolded": ["exchangeratecalculator", "reportdailybasetransaction"],
      "branch": "codegen/Reporting"
    },
    "Funds": {
      "planned_slices": ["withdrawaldeclinenotified"],
      "scaffolded": ["withdrawaldeclinenotified"],
      "branch": "codegen/Funds"
    }
  },
  "total_planned": 3,
  "total_scaffolded": 3
}
```

## Phase 2: AI Business Logic

Claude Code CLI runs inside CI for each context. It fills in the parts that require reasoning:

### What Claude Code Does

1. **Reads `TODO_CONTEXT.md`** — scaffold generates this per-slice file with implementation guidance, field mappings, spec scenarios, and event dependencies.

2. **Reads `learned_hints.md`** — accumulated domain patterns from previously implemented slices (predicates, computed fields, test data patterns).

3. **Fills `gwts.ts` predicates** — replaces `return false` stubs with real business logic:
   ```typescript
   // Before (scaffold)
   export const hasInsufficientEffectiveFunds = (...): boolean => false;

   // After (Claude Code)
   export const hasInsufficientEffectiveFunds = (
     balance: number, command: ApproveWithdrawal
   ): boolean => balance < command.amount;
   ```

4. **Fills `commandHandler.ts`** — branching logic and computed fields:
   ```typescript
   // Before (scaffold)
   // TODO: implement command handler logic

   // After (Claude Code)
   if (hasInsufficientEffectiveFunds(balance, command)) {
     stream.appendEventFundsWithdrawalDeclined({ ... });
   } else {
     stream.appendEventFundsWithdrawalApproved({ ... });
   }
   ```

5. **Runs tests** — verifies each slice's tests pass before moving to the next.

### Verification

After Claude Code finishes:

- **`evdb-verify`** validates generated code matches the spec contract (field names/types, event count, predicate count, handler structure)
- **`npm run test:slices`** runs all slice tests across the project

---

## Phase 3: Commit & PR

One PR per business context:

- **Branch:** `codegen/<Context>` (e.g. `codegen/Reporting`)
- **Commit message:** includes list of slices and pipeline steps
- **PR body:** summary, pipeline steps, review checklist

### PR Review Checklist

- [ ] Predicate logic in `gwts.ts` matches domain rules
- [ ] Command handler branching covers all spec scenarios
- [ ] Generated field computations are correct
- [ ] Tests pass and cover edge cases

## Triggers

| Trigger | When | Gate |
|---------|------|------|
| **Auto (push)** | Push to `main` that changes `.eventmodel/config.json` or `.eventmodel2/config.json` | Content hash check — skips if only status/index fields changed |
| **Manual** | `workflow_dispatch` from GitHub Actions tab | No gate — always runs |

### What Does NOT Trigger the Pipeline

| Scenario | Why |
|----------|-----|
| Editing business code in `src/` | Not in path filter |
| evdb-diff updating `index.json` statuses | `index.json` not in path filter |
| Editing individual `slice.json` files | Only `config.json` triggers |
| Changing `config.json` status/index fields only | Content hash gate detects no real change |

| File | Role |
|------|------|
| `.claude/scripts/split_config.py` | Splits single Miro JSON into per-context/slice files |

| Skill | Script | Purpose |

## Local Testing

### With `act` (Docker-based GitHub Actions runner)

```bash
# Install act
brew install act

# Create .secrets file with API key
echo "ANTHROPIC_API_KEY=sk-ant-api03-..." > .secrets

# Run the local test workflow
act workflow_dispatch \
  -W .github/workflows/generate-slices-local.yml \
  --input em_dir=.eventmodel2 \
  --secret-file .secrets \
  --container-architecture linux/amd64
```

### Without Docker (direct local run)

```bash
# Phase 1: Scaffold
python3 .claude/scripts/generate_slices.py --root . --em-dir .eventmodel2

# Phase 2: Claude Code fills business logic
export ANTHROPIC_API_KEY=sk-ant-api03-...
claude --print --dangerously-skip-permissions --max-turns 50 \
  "Context: Reporting. Slices: exchangeratecalculator,reportdailybasetransaction.
   Read each slice's TODO_CONTEXT.md and fill in business logic."
```

## Required Secrets

| Secret | Where | Purpose |
|--------|-------|---------|
| `ANTHROPIC_API_KEY` | GitHub repo settings -> Secrets | Claude Code API access for Phase 2 |
| `GITHUB_TOKEN` | Auto-provided by GitHub Actions | Creating branches and PRs |

### Setting up the API key

1. Go to **console.anthropic.com**
2. **Settings -> API Keys -> Create Key**
3. In your GitHub repo: **Settings -> Secrets and variables -> Actions -> New repository secret**
4. Name: `ANTHROPIC_API_KEY`, Value: your key
