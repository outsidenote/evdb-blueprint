What this pipeline does

This CI pipeline turns an event model diagram (exported from Miro as JSON) into production-ready TypeScript code — automatically. It scaffolds event-sourced CQRS slices, fills business logic with AI, verifies the output against contracts, self-heals failures, and opens a pull request with a confidence score and full audit trail.

A developer implementing 6 slices manually — writing command handlers, predicates, projection SQL, tests, wiring adapters — takes 3 to 5 days. This pipeline does it in ~15 minutes, with every decision auditable and every pattern learned for next time. The system routes simple slices through a fast path and complex slices through a deep path, so each slice gets exactly the attention it needs — no more, no less.

Phases

Phase 0 — Plan

Deterministic ▸

Everything before AI runs. Produces the matrix of contexts and per-slice decisions. Every step is deterministic — no AI calls, no network dependencies beyond GitHub.

Detect + Gate

stages/detect.py

Computes an MD5 hash of config.json after stripping volatile fields (status, index, lastModified). If the normalized hash matches the previous commit, the entire pipeline is skipped — no wasted compute on cosmetic config edits.

  • Priority: explicit --em-dir flag > git diff detection > default .eventmodel
  • workflow_dispatch always passes the gate (manual trigger = intent to run)
  • Supports --mode simulate for dry-run testing

Inputs

  • config.json push event
  • github.event.before SHA
  • --em-dir (optional override)

Outputs

  • em_dir — .eventmodel or .eventmodel2
  • changed — true / false
  • mode — live / simulate

Scaffold

stages/scaffold.py

A 5-step deterministic pipeline that turns a Miro-exported config.json into scaffolded TypeScript source files with normalized specs.

1. Splitsplit_config.pySplits monolithic config.json into per-slice JSON files under .eventmodel/.slices/
2. Diffevdb-diffAudits statuses against actual source code. Updates each slice to Planned (no code) or Review (code exists)

Planned and Created always. Done and Review only if their expected source files are missing on disk (guards against someone deleting generated code).

Risk Score

stages/risk_score.py

Deterministic structural analysis. No AI calls. Produces a 0.0–1.0 weighted risk score per slice from 5 factors. Each factor is normalized to 0.0–1.0 (where 1.0 = maximum risk), then multiplied by its weight. The weights sum to 1.0, so the final score is also 0.0–1.0. Every factor is backed by real data — no stubs or hardcoded defaults.

Formula

risk = ∑(factor.normalized × factor.weight)

FactorWeightSignalNormalization
Spec complexity0.35specifications × eventsmin(raw / 12, 1.0)
Slice type0.20STATE_CHANGE 0.2, QUERY 0.1, PROCESSOR 0.6, ENRICHMENT 0.5Fixed lookup
Blast radius0.20Number of events emitted (proxy for downstream consumers)min(events / 10, 1.0)
Field types0.15Proportion of risky types (Custom, List, DateTime)risky_count / total_fields
TODO density0.10Placeholder count in scaffolded .ts filesmin(count / 10, 1.0)
Why only 5 factors?
Earlier versions included prior_failure_rate (weight 0.20) and spec_novelty (weight 0.10), but both returned a hardcoded 0.5 for every slice — 30% of the score was dead weight that differentiated nothing. They were removed. When real run history accumulates (from slice-metrics.jsonl), prior failure rate can be added back with actual data behind it.

Score 0.07 — the lowest in the Portfolio context. The risk score sees almost nothing: 0 specs, 1 event, mostly String/Double fields. But this slice contains an 8-step Basel III risk engine with Monte Carlo simulation. The risk score is correct — it measures structural difficulty for code generation (branching, dependents, types), not algorithmic complexity. The AI will still use Opus with self-test for this slice, and the complexity of the algorithm will drive the number of turns used.

Structural risk vs algorithmic complexity
The risk score measures how hard the code generation structure is: how many branches (specs × events), how many things break if wrong (blast radius), how tricky the types are (DateTime, Custom). It does not measure how hard the algorithm is — that shows up in the number of turns the AI needs. A slice can score 0.07 (LOW) on structural risk but still need 18 turns because the algorithm is a Monte Carlo simulation.

Phase 1 — Implement

AI-Assisted Learning ▸

One job per context via matrix strategy. Claude fills TODO placeholders in scaffolded files. Slices run in parallel via ThreadPoolExecutor (max 3 concurrent). Every AI slice uses Opus with a self-test protocol — the AI runs all available unit tests within its session and self-corrects before handing off. Hints from prior runs and clarification answers from PR comments are injected into prompts. All commits are SSH-signed by the bot account.

Deterministic Trivial Fast-Path

implement_slice.py → scan_todos()

Before calling any AI, each slice is scanned for TODO markers in its .ts files. If only generic scaffold TODOs remain (no real business logic needed), the slice takes a zero-cost deterministic path.

PathConditionAction
SKIPOnly generic TODOs and no spec descriptionsStrip TODO comments deterministically — $0, instant
AIReal TODOs or spec has logic (descriptions, specifications)Opus, 20 turns, self-test with exact test commands
Why no FAST/DEEP split?
On Max plan, Sonnet and Opus cost the same. The Haiku classification step added ~2s per slice to route between models with no cost benefit. Now every AI slice gets Opus with self-test — simpler, faster, and the same quality for every slice.

Prompt Construction

implement_slice.py → build_deep_prompt()

The deep-mode prompt includes key conventions (pure handlers, appendEvent syntax, .js extensions, view state access), then injects retrieved hints, followed by the full TODO spec. Example hint block injected into the prompt:

## Relevant Hints
- Predicate `hasInsufficientBalance`: `balance < command.amount + command.commission` (from: withdrawfunds)
- Computed field `reason` = `template literal with balance interpolation` (from: withdrawfunds)
- Projection SQL: `INSERT INTO ... jsonb_build_object('balance', COALESCE(...) + $5)` (from: accountbalancereadmodel)

Self-Test Protocol

implement_slice.py → build_prompt()

After filling all TODOs, the AI runs every unit test that exists in its CWD. Exact node --import tsx --test commands are embedded in the prompt — no guessing. If tests fail, the AI reads the assertion error, fixes its code, and re-runs — all with full context still alive.

Self-tests (pure unit tests)NOT self-tested (needs infrastructure)
tests/command.slice.test.ts — predicates, branching, error messagesprojection.slice.test.ts — needs PostgreSQL via testcontainers
tests/projection.test.ts — SQL shape, params, structureCross-slice integration tests
tests/enrichment.test.ts — computation logic, formulas

Stopping rules:

  • Tests pass → done
  • Same test fails twice with the same error → stop (issue may be structural)
  • Failure is caused by imports, scaffold, or missing dependencies → stop
  • Turn budget reached → stop
Why self-test works
The biggest win is not fewer sessions. It is that the model that introduced the bug is the same model that sees the exact failing assertion and fixes it — while its full working context is still alive. The repair ladder starts cold with a truncated error message. Self-test has everything.

Clarification Answers Injection

generate-slices.yml → "Load clarification answers"

If a previous run posted clarification questions on the PR, and a developer answered in comments, those answers are fetched and injected into the prompt. The flow:

  • Workflow step fetches human comments from the most recent PR for this context
  • Comments are written to /tmp/pr-clarification-answers.txt
  • Prompt appends them as ## Clarification from Developer
  • AI uses the answers to resolve ambiguity in the spec

Reuse Feedback

implement_slice.py → _record_reuse_feedback()

After all workers complete, the main thread collects which hint record IDs were used by successful AI runs (turns > 0). For each used hint, it increments times_reused and sets last_reused_at. Written to a separate artifact — never mutates the original discovery file.

Trigger

  • Phase 2 complete (all workers done)
  • At least one hint was used by a successful AI run

Output

  • /tmp/learned-discoveries.updated.json
  • Original records shallow-copied, reuse fields updated

Phase 2 — Verify + Repair

Deterministic AI if needed ▸

Verify + Test

stages/verify_and_test.py

Three-step verification pipeline. Each step must pass for the slice to be considered successful.

StepToolWhat it checksTimeout
1. Normalizeevdb-normalize --allSpec data is structurally valid—
2. Verifyevdb-verify --all --jsonGenerated code matches normalized spec contracts—
3. Testnode --testUnit tests pass: command, projection, view, enrichment120s per file

Test patterns searched:

  • slices/*/tests/command.slice.test.ts — command handler unit tests
  • slices/*/tests/projection.test.ts — projection shape tests
  • slices/*/projection.slice.test.ts — projection integration tests (real PostgreSQL via testcontainers)
  • endpoints/*/tests/enrichment.test.ts — enrichment logic tests

Pass logic: verify_passed AND test_passed. Both must be true.

Results are written as structured JSON to /tmp/test-results.json — each entry has file, slice, passed, exit_code, error. The classifier reads this directly instead of parsing test output text.

ESLint

verify_and_test.py → run_lint()

Runs ESLint on only the .ts files that changed (via git diff --name-only HEAD~1). Excludes test files. Enforces zero warnings — catches any types, unused imports, and other issues the AI might introduce. Failures are appended to /tmp/test-results.json as lint_error class so the repair ladder can fix them.

Classify Failures

stages/classify.py

Deterministic classification from structured test results JSON — no regex parsing of test output. Each failed test already has its slice name and error text. The classifier checks for flaky patterns (ETIMEOUT, ENOMEM) and bins the rest as the appropriate failure class. Includes lint_error for ESLint violations.

It receives the spec (TODO_CONTEXT.md), the failing test file, and the test output. The key instruction: "the test is derived from the spec and is CORRECT — fix the implementation to match."

CWD resolution

Repair picks the directory that contains the files to fix — not just slices/. For enrichment failures, it finds endpoints/AssessLoanRisk/ where enrichment.ts lives. For projection failures, it finds slices/PortfolioSummary/ where index.ts lives.

Lessons learned
Five root causes prevented repair from working in early runs: (1) Budget too low ($0.25 → $2.00). (2) No spec context → now includes TODO_CONTEXT.md. (3) enrichment.ts not in allowed_files → added. (4) index.ts not in allowed_files → added. (5) Wrong CWD for endpoint slices → now finds correct directory. All five are now fixed.

Phase 3 — Learn

Learning Deterministic ▸

Only fires after verify + test passes. Extracts reusable domain patterns from the AI-generated code and writes them as structured memory records. No AI involved — pure regex parsing and deterministic fingerprinting.

stages/learn.py → extract_*_patterns()

Developer Workflow

Three options to resolve:

  • Update Miro — clarify the spec description, re-export config.json, push → pipeline re-triggers with clearer spec
  • Comment on PR — answer the question directly. On re-trigger, the pipeline reads PR comments and injects them as ## Clarification from Developer in the AI prompt
  • Fix manually — if it is faster to change one line than re-run the pipeline

Spec Quality Metrics

Each slice metric record in slice-metrics.jsonl includes clarification_needed: true/false. Over time this reveals:

  • Which slices/specs generate the most ambiguity
  • Which failure classes recur most often
  • Whether Miro modeling conventions need tightening

Learning Loop

Learning ▸

The structured memory system forms a closed feedback loop across CI runs. Patterns extracted from successful implementations are reused to improve future implementations.

End-to-End Flow

Run N (current):
  1. implement_slice.py loads /tmp/learned-discoveries.json
  2. build_memory_index() creates inverted indexes (O(N), once)
  3. Per slice: retrieve() narrows via index, scores, returns top hints
  4. Hints injected into prompt as "## Relevant Hints" block
  5. Claude fills TODOs with domain logic
  6. _record_reuse_feedback() bumps times_reused for used hints
     → writes /tmp/learned-discoveries.updated.json

  7. verify_and_test.py runs (pass/fail)
  8. If passed: learn.py extracts new patterns from the AI output
     → writes /tmp/learned-discoveries.json

Run N+1 (next):
  1. implement_slice.py loads the updated discoveries
  2. New patterns + reuse metadata available for retrieval
  3. Frequently reused patterns score higher (+1 recently_used)
  4. Never-reused patterns are candidates for deprecation

Future: Vector DB Upgrade Path

The current system is local JSON with deterministic scoring. To upgrade to Chroma or LanceDB:

  • Replace the score_record() loop with a similarity search
  • Use content.logic / content.sql_excerpt as embedding text
  • Keep the same metadata filters (type, context, status, file)
  • Keep format_for_prompt() unchanged — it formats whatever comes back
  • Add a --backend flag: local (current) vs chroma vs lancedb
  • The retrieve() signature stays the same — callers don't change

The multiplier effect
This is for one context (Portfolio) with 6 slices. A real event model may have 3–5 contexts with 20+ slices. Manually, that is weeks of developer time. What the developer's role becomes

The pipeline doesn't eliminate the developer. It changes their job from writing boilerplate to reviewing AI output. The confidence score and review guide tell them exactly where to look:

  • HIGH confidence (85+) — skim the diff, approve if it looks right
  • MEDIUM confidence (50–79) — review guide flags specific files (e.g., "gwts.ts was AI-repaired, check predicate logic")
  • LOW confidence (<50) — manual inspection required, but the pipeline already did the scaffolding and wiring

ESLint Enforcement

Zero-warning ESLint policy on AI-generated code. Only changed .ts files are linted (via git diff). The lint_error failure class in the classifier allows the repair ladder to fix violations like any types.

Date Serialization Convention

JavaScript Date objects serialize differently through the pg driver vs JSON.stringify(). The scaffold template instructs the AI: "convert Date fields to .toISOString() before passing as SQL params." This ensures consistent ISO format in PostgreSQL jsonb and matching test assertions.

Projection SQL Type Casts

Parameters inside jsonb_build_object() must have explicit type casts ($3::text, $4::numeric). PostgreSQL cannot infer types in jsonb context. The scaffold template shows this pattern and the integration tests catch violations by executing SQL against real PostgreSQL.
